# Phase 5: Client UTM Capture Hook - Context

**Gathered:** 2026-04-25
**Status:** Ready for planning

<domain>
## Phase Boundary

Deliver the browser-side attribution pipeline: a `useAttribution` custom hook that captures UTM parameters on first mount, generates/persists the `mvp_vid` anonymous visitor ID in localStorage, pings the server session endpoint, classifies referrer-based traffic sources, exposes `visitorId` to other components, and wires client-side conversion calls for phone clicks and booking starts.

Done when: the full round-trip is live — a visitor arriving with `?utm_source=google` causes a `visitor_sessions` row to be created, and the lead form submission passes `visitorId` so `form_leads.visitor_id` is populated.

No new server routes. No UI changes. No admin section work. All changes are in `client/src/`.

</domain>

<decisions>
## Implementation Decisions

### Hook Structure

- **D-01:** Create `client/src/hooks/use-attribution.ts` — a custom hook following the existing `useSEO`/`useAuth` pattern. Returns `{ visitorId: string | null }` so components that need visitorId can import and call the hook.
- **D-02:** The hook is called ONCE in `App.tsx` inside the `AppContent` component (the inner component that already has `useLocation`, `useEffect`, analytics init, etc.). Do NOT call it in route components or page components — only at the App root to avoid re-mounting.
- **D-03:** The hook uses TWO separate `useEffect` calls:
  1. `useEffect(() => { /* capture UTMs + generate mvp_vid + first ping */ }, [])` — runs once on mount. Reads `window.location.search` BEFORE any navigation destroys it. Generates or retrieves `mvp_vid` from localStorage. Classifies referrer. POSTs to `/api/attribution/session` with the full payload including ft_* UTMs.
  2. `useEffect(() => { /* update lt_* on each navigation */ }, [location])` — runs on every Wouter route change. POSTs to `/api/attribution/session` with only `{ visitorId, ltLandingPage: location, ltSource, ltMedium }`. Server upsert never overwrites `ft_*` columns so this is safe.

### Visitor ID (mvp_vid)

- **D-04:** Key name: `mvp_vid` in `localStorage`. Distinct from `formLeads.sessionId` (a per-form UUID generated by the form itself).
- **D-05:** If `mvp_vid` exists in localStorage, use it. If not, generate with `crypto.randomUUID()` and store it before the first session ping.
- **D-06:** `visitorId` is stored in React state (`useState`) so components can read it reactively. The hook returns `{ visitorId }`.

### UTM Capture

- **D-07:** Read UTMs from `new URLSearchParams(window.location.search)` inside the empty-deps `useEffect`. Parameters: `utm_source`, `utm_medium`, `utm_campaign`, `utm_term`, `utm_content`, `utm_id`. Normalize: `.toLowerCase().trim()`.
- **D-08:** If NO UTM params present, classify from `document.referrer`:
  - `document.referrer` empty → `sourceChannel: 'Direct'`, `ftSource: '(direct)'`, `ftMedium: '(none)'`
  - Referrer host matches search engines (google, bing, yahoo, duckduckgo, baidu, yandex) → `sourceChannel: 'Organic Search'`, `ftSource: host`, `ftMedium: 'organic'`
  - Referrer host matches social (facebook.com, instagram.com, youtube.com, tiktok.com, linkedin.com, twitter.com, x.com, pinterest.com) → `sourceChannel: 'Social Media'`, `ftSource: host`, `ftMedium: 'social'`
  - Any other referrer → `sourceChannel: 'Referral'`, `ftSource: host`, `ftMedium: 'referral'`
  - No referrer, no UTM → `sourceChannel: 'Direct'`
- **D-09:** When UTM params ARE present: `sourceChannel` is derived from `utm_medium`:
  - `cpc` | `ppc` | `paid` | `paidsearch` | `paidsocial` → `'Paid Ads'`
  - `social` | `social-media` → `'Social Media'`
  - `email` → `'Email'`
  - `organic` → `'Organic Search'`
  - Any other → `'Referral'`

### Session Ping

- **D-10:** Both pings use `navigator.sendBeacon('/api/attribution/session', blob)` with fallback to `fetch(..., { keepalive: true })` — same pattern as `reportEventHit` in `analytics.ts`. Never `await`, never block the critical path.
- **D-11:** Ping body shape matches `InsertVisitorSession` from `shared/schema.ts` (without `id`, `firstSeenAt`, `lastSeenAt`). The server's upsert handles the rest.
- **D-12:** Device type detection: `navigator.userAgent` checked for mobile/tablet/desktop. Simple string match: `/(iPhone|Android.*Mobile|BlackBerry)/i` → `'mobile'`, `/(iPad|Android(?!.*Mobile))/i` → `'tablet'`, else → `'desktop'`.

### Form visitorId Injection

- **D-13:** `LeadFormModal.tsx` already calls `useAttribution()` to get `visitorId`, then includes it in the `payload` object before the `fetch("/api/form-leads/progress", ...)` call (line ~485). No need to scatter `localStorage.getItem` calls — the hook is the single source.
- **D-14:** `visitorId` is optional in the form payload (server already accepts it as optional). If `useAttribution` hasn't resolved yet (null), the form sends without it — no blocking.

### Page View Events (CONV-05)

- **D-15:** In the existing `useEffect(() => { trackPageView(location); }, [location])` block in App.tsx, also call `reportAttributionPageView(location, visitorId)` — a small helper in the attribution hook that POSTs `{ eventName: 'page_view', pagePath: location, visitorId }` to `/api/analytics/hit` via sendBeacon. Only fires when `visitorId` is non-null.
- **D-16:** Do NOT add `page_view` to `SERVER_REPORTED_EVENTS` in `analytics.ts` — that set is for generic analytics events. Attribution page view tracking is a separate concern handled by the hook.

### Phone Click Conversion (CONV-02)

- **D-17:** In `Navbar.tsx`, extend the existing phone link `onClick` handler to also fire `POST /api/attribution/conversion` with `{ visitorId, conversionType: 'phone_click', pagePath: window.location.pathname }` via sendBeacon. `visitorId` comes from a shared `getVisitorId()` utility (reads `localStorage.getItem('mvp_vid')` directly — simpler than prop-drilling the hook return value into Navbar).
- **D-18:** Create `client/src/lib/attribution.ts` — a thin utility module with: `getStoredVisitorId(): string | null`, `fireConversionEvent(type, pagePath?): void`. Used by Navbar and the booking flow without needing to call `useAttribution()` (hook call at non-root level is fine but adds overhead; the utility avoids it).

### Booking Start Conversion (CONV-04)

- **D-19:** Find where the booking flow starts in the client (likely a "Book Now" or "Start Booking" button) and call `fireConversionEvent('booking_started')` on click, using the utility from `attribution.ts`.
- **D-20:** This is a best-effort fire — if `mvp_vid` is null (hook hasn't run or localStorage cleared), the conversion event is silently skipped.

### Claude's Discretion

- Exact location of booking start CTA in client code — executor reads the codebase to find it
- Whether to memoize the classification function — Claude's call
- Error handling granularity for failed session pings — silent catch is sufficient

</decisions>

<specifics>
## Specific Ideas

- The `useAttribution` hook should NOT call `useLocation()` itself — the parent `AppContent` already has `[location]` and can pass it as a prop or the hook can call `useLocation()` directly (both are fine; hook calling it is cleaner).
- The two-useEffect pattern (mount + navigation) avoids a common pitfall where the UTM capture fires on every navigation and reads an empty URL after the first route change.
- `fireConversionEvent` in `attribution.ts` uses sendBeacon so it never blocks the thread on phone click.

</specifics>
