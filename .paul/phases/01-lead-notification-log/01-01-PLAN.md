<objective>
## Goal
Add a `notification_logs` table to the database and expose storage-layer CRUD methods so any outgoing notification (Twilio SMS, Resend email, GoHighLevel sync) can later be recorded with recipient, preview, status and provider message ID.

## Purpose
Today notifications are fire-and-forget â€” no audit trail exists. This plan lays the data foundation; subsequent plans (01-02 through 01-04) wire up the integrations, API, and admin UI on top of this table.

- New `notificationLogs` table + Drizzle types + Zod insert schema in `shared/schema.ts`
- New storage methods (`createNotificationLog`, `getNotificationLogsByLead`, `listNotificationLogs`) on `IStorage` and `DatabaseStorage` in `server/storage.ts`
- Migration applied via `npm run db:push`
- Typecheck clean (`npm run check`)
</objective>

  <action>
    In `shared/schema.ts`, after the `formLeads` table definition (around line 215), add:

    ```ts
    export const notificationLogs = pgTable("notification_logs", {
      id: serial("id").primaryKey(),
      leadId: integer("lead_id").references(() => formLeads.id, { onDelete: "set null" }),
      channel: varchar("channel", { length: 20 }).notNull(), // 'sms' | 'email' | 'ghl_sync'
      trigger: varchar("trigger", { length: 40 }).notNull(), // 'lead_completed' | 'lead_abandoned' | 'new_chat' | 'low_performance'
      recipient: varchar("recipient", { length: 255 }).notNull(),
      recipientName: varchar("recipient_name", { length: 100 }),
      subject: varchar("subject", { length: 255 }),
      preview: text("preview").notNull(),
      status: varchar("status", { length: 20 }).notNull(), // 'sent' | 'failed' | 'skipped'
      errorMessage: text("error_message"),
      providerMessageId: varchar("provider_message_id", { length: 100 }),
      metadata: jsonb("metadata"),
      sentAt: timestamp("sent_at").defaultNow().notNull(),
    }, (t) => ({
      leadIdx: index("notification_logs_lead_idx").on(t.leadId),
      sentAtIdx: index("notification_logs_sent_at_idx").on(t.sentAt),
      channelIdx: index("notification_logs_channel_idx").on(t.channel),
      statusIdx: index("notification_logs_status_idx").on(t.status),
    }));
    ```

    Then in the SCHEMAS / TYPES sections lower in the file, following the existing convention used for other tables, export:
    ```ts
    export const insertNotificationLogSchema = createInsertSchema(notificationLogs).omit({
      id: true,
      sentAt: true,
    });
    export type NotificationLog = typeof notificationLogs.$inferSelect;
    export type InsertNotificationLog = z.infer<typeof insertNotificationLogSchema>;
    ```

    Import `varchar` from `drizzle-orm/pg-core` in the top-level import statement if not already imported. Keep `integer`, `index`, `jsonb`, `pgTable`, `serial`, `text`, `timestamp` imports unchanged.

    - Do NOT add a FK cascade delete â€” use `set null` so logs survive lead deletion for audit purposes.
    - Do NOT add any columns not listed (no `readAt`, no `retryCount` â€” out of scope for v1.1).
  </action>
